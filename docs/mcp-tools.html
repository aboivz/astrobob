MCP Tools Reference

Complete reference for the 5 Model Context Protocol (MCP) tools that AstroBob exposes to IBM Bob and other MCP clients.

What is MCP? The Model Context Protocol allows AI agents like Bob to access external tools and data sources. AstroBob implements an MCP server that Bob can connect to for persistent memory operations.

Tool Overview

Tool Purpose When to Use
remember Store new memory After learning something new or completing a task
recall Search memories Before starting a task or when needing context
reflect Create procedure from episodes When discovering reusable patterns
forget Delete memory When information becomes outdated
audit_trail View memory history When debugging or understanding provenance

Connecting Bob to AstroBob

1. Start the MCP Server

# In your project directory
astrobob mcp

2. Configure Bob

The astrobob init command creates .bob/mcp.json:

{
  "mcpServers": {
    "astrobob": {
      "command": "astrobob",
      "args": ["mcp"],
      "env": {
        "ASTRA_DB_API_ENDPOINT": "https://xxx.apps.astra.datastax.com",
        "ASTRA_DB_APPLICATION_TOKEN": "AstraCS:xxxxx"
      }
    }
  }
}

3. Use in Bob

Open Bob in Advanced mode to access MCP tools. Bob will automatically discover the 5 AstroBob tools.

remember

Store a new memory in AstroBob's persistent memory system.

When to Use

  • ✅ After learning a project convention or fact
  • ✅ After completing a task or fixing a bug
  • ✅ When discovering a reusable procedure
  • ✅ After making an important decision
  • ❌ For transient state or temporary data
  • ❌ For information already stored

Input Schema

{
  "content": "string (required)",
  "memory_type": "semantic | episodic | procedural (required)",
  "project": "string (optional, defaults to current directory)",
  "tags": ["string"] (optional),
  "importance": 1-5 (optional, default: 3),
  "confidence": 0.0-1.0 (optional, default: 1.0),
  "source": "bob | bobshell | wxo | cli | user (optional, default: bob)"
}

Parameters

Parameter Type Required Description
content string Yes The memory content (what to remember)
memory_type enum Yes Type: semantic (facts), episodic (events), procedural (how-tos)
project string No Project name (defaults to current directory)
tags array No Tags for categorization (e.g., ["api", "auth"])
importance integer No 1-5 scale (5=critical, 4=important, 3=useful, 2=nice-to-know, 1=trivial)
confidence float No 0.0-1.0 confidence level (1.0=certain, 0.5=uncertain)
source enum No Source of memory (defaults to "bob")

Return Value

{
  "success": true,
  "memory_id": "01HXXX...",
  "message": "Memory stored successfully"
}

Examples

Semantic Memory (Fact)
remember({
  "content": "This project uses FastAPI 0.104 with async routes and Pydantic v2 models",
  "memory_type": "semantic",
  "importance": 4,
  "tags": ["fastapi", "architecture", "dependencies"]
})
Episodic Memory (Event)
remember({
  "content": "Fixed authentication bug by adding token refresh logic in auth.py. The issue was expired tokens not being handled gracefully.",
  "memory_type": "episodic",
  "importance": 3,
  "tags": ["bugfix", "auth", "tokens"]
})
Procedural Memory (How-To)
remember({
  "content": "To add a new API endpoint: 1) Define route in routes/ with @router decorator, 2) Add request/response schemas in schemas/, 3) Write unit tests in tests/api/, 4) Update OpenAPI docs, 5) Test with curl",
  "memory_type": "procedural",
  "importance": 5,
  "tags": ["api", "procedure", "development"]
})

Best Practices

  • Be specific: "FastAPI uses async def for routes" beats "FastAPI is async"
  • Include context: Mention file names, error messages, or specific values
  • Use importance wisely: Reserve 5 for critical patterns you'll reuse often
  • Tag consistently: Use lowercase, hyphenated tags like "api-design"
  • Avoid duplication: Check with recall() first

recall

Search and retrieve memories using natural language queries with hybrid search (vector + keyword + reranking).

When to Use

  • ✅ Before starting a non-trivial task
  • ✅ When you need project context
  • ✅ To check if a problem was solved before
  • ✅ To find relevant procedures
  • ❌ For every single action (use judgment)

Input Schema

{
  "query": "string (required)",
  "memory_type": "semantic | episodic | procedural | all (optional)",
  "project": "string (optional)",
  "tags": ["string"] (optional),
  "limit": 1-50 (optional, default: 10),
  "min_importance": 1-5 (optional, default: 1)
}

Parameters

Parameter Type Required Description
query string Yes Natural language search query
memory_type enum No Filter by type (defaults to "all")
project string No Filter by project (defaults to current)
tags array No Filter by tags (AND logic)
limit integer No Maximum results (1-50, default: 10)
min_importance integer No Minimum importance level (1-5)

Return Value

{
  "success": true,
  "results": [
    {
      "memory_id": "01HXXX...",
      "memory_type": "procedural",
      "content": "To add API endpoint: ...",
      "score": 0.92,
      "importance": 5,
      "tags": ["api", "procedure"],
      "created_at": "2026-05-15T10:30:00Z",
      "access_count": 12
    }
  ],
  "total": 1
}

Query Routing

AstroBob automatically routes queries to the most relevant memory types based on keywords:

Query Pattern Priority Order Example
"how to", "should i", "when" Procedural → Semantic → Episodic "how to add API endpoint"
"what is", "is", "does" Semantic → Procedural → Episodic "what is the API rate limit"
"last time", "why did", "when did" Episodic → Semantic → Procedural "last time we fixed auth bug"
Other Procedural → Semantic → Episodic "authentication error"

Examples

General Search
recall({
  "query": "how to add API endpoint"
})

// Returns procedural memories first
Type-Specific Search
recall({
  "query": "authentication",
  "memory_type": "episodic",
  "tags": ["bugfix"]
})

// Returns only episodic memories tagged with "bugfix"
High-Importance Only
recall({
  "query": "architecture decisions",
  "min_importance": 4,
  "limit": 5
})

// Returns top 5 important memories

Ranking Algorithm

Results are ranked using a weighted formula:

final_score = 
  0.55 × rerank_score +
  0.15 × (importance / 5) +
  0.15 × recency_factor +
  0.10 × success_factor -
  0.05 × staleness_penalty

reflect

Create a procedural memory (reusable procedure) from one or more episodic memories (past experiences).

When to Use

  • ✅ After solving similar problems multiple times
  • ✅ When discovering a reusable pattern
  • ✅ To distill lessons from experiences
  • ✅ Weekly reflection on what you learned
  • ❌ For every single episode (be selective)

Input Schema

{
  "project": "string (required)",
  "episode_ids": ["string"] (required, 1-10 IDs),
  "lesson": "string (required)",
  "tags": ["string"] (optional),
  "importance": 1-5 (optional, default: 4)
}

Parameters

Parameter Type Required Description
project string Yes Project name
episode_ids array Yes 1-10 episodic memory IDs to reflect on
lesson string Yes The procedural lesson learned
tags array No Tags for the new procedure
importance integer No Importance level (default: 4)

Return Value

{
  "success": true,
  "procedural_memory_id": "01HYYY...",
  "derived_from": ["01HXXX1...", "01HXXX2..."],
  "message": "Reflection created successfully"
}

Example

Creating a Reflection
// First, find related episodes
recall({
  "query": "fixed authentication bug",
  "memory_type": "episodic"
})

// Then create reflection
reflect({
  "project": "myapp",
  "episode_ids": ["01HXXX1...", "01HXXX2...", "01HXXX3..."],
  "lesson": "When fixing auth bugs: 1) Check token expiry first, 2) Verify refresh logic, 3) Add integration tests, 4) Update error messages",
  "tags": ["auth", "debugging", "testing"],
  "importance": 4
})

Workflow

  1. Use recall() to find related episodic memories
  2. Identify common patterns across episodes
  3. Distill into a step-by-step procedure
  4. Call reflect() with episode IDs and lesson
  5. New procedural memory is created with provenance tracking

forget

Soft-delete a memory (marks as deleted but preserves for audit trail).

When to Use

  • ✅ When information becomes outdated
  • ✅ When a procedure is superseded
  • ✅ To clean up incorrect memories
  • ❌ For temporary cleanup (memories are already filtered)

Input Schema

{
  "memory_type": "semantic | episodic | procedural (required)",
  "memory_id": "string (required)"
}

Parameters

Parameter Type Required Description
memory_type enum Yes Type of memory to delete
memory_id string Yes ULID of the memory

Return Value

{
  "success": true,
  "message": "Memory marked as deleted"
}

Example

forget({
  "memory_type": "semantic",
  "memory_id": "01HXXX..."
})
⚠️ Note: This is a soft delete. The memory is marked with deleted_at timestamp but not removed from the database. It won't appear in recall() results but remains in the audit trail.

audit_trail

View complete history and metadata for a memory.

When to Use

  • ✅ Debugging memory issues
  • ✅ Understanding memory provenance
  • ✅ Checking access patterns
  • ✅ Verifying memory lifecycle

Input Schema

{
  "memory_type": "semantic | episodic | procedural (required)",
  "memory_id": "string (required)"
}

Parameters

Parameter Type Required Description
memory_type enum Yes Type of memory
memory_id string Yes ULID of the memory

Return Value

{
  "success": true,
  "audit": {
    "memory_id": "01HXXX...",
    "memory_type": "procedural",
    "content": "To add API endpoint: ...",
    "created_at": "2026-05-15T10:30:00Z",
    "updated_at": "2026-05-15T12:00:00Z",
    "last_accessed_at": "2026-05-15T14:22:00Z",
    "access_count": 12,
    "success_count": 8,
    "importance": 5,
    "tags": ["api", "procedure"],
    "provenance": {
      "derived_from": ["01HYYY...", "01HZZZ..."],
      "session_id": "sess_abc123",
      "source": "bob"
    },
    "exported_as_skill": ".bob/skills/learned/add-api-endpoint/SKILL.md",
    "exported_at": "2026-05-15T14:00:00Z"
  }
}

Example

audit_trail({
  "memory_type": "procedural",
  "memory_id": "01HXXX..."
})

Decision Tree for Tool Usage

When Bob Should Use Each Tool
Starting a task?
├─ Yes → recall() to check for relevant context
│  ├─ Found procedure? → Follow it
│  └─ Not found? → Proceed and remember() when done
└─ No → Continue

Completed a task?
├─ Learned something new? → remember() as semantic
├─ Fixed a bug? → remember() as episodic
└─ Discovered a pattern? → remember() as procedural

Solved similar problems 2-3 times?
└─ Yes → reflect() to create reusable procedure

Information outdated?
└─ Yes → forget() the old memory

Need to understand memory history?
└─ Yes → audit_trail() for full details

Best Practices

💡 For Bob Users

  • Recall before acting: Check for procedures before starting non-trivial tasks
  • Remember after learning: Store important discoveries immediately
  • Reflect regularly: Weekly reflections turn experience into reusable knowledge
  • Be specific: Detailed memories are more useful than generic ones
  • Use importance wisely: Reserve 5 for critical patterns

🔧 For Developers

  • Test with MCP Inspector: Use astrobob mcp --http for debugging
  • Monitor access patterns: Use audit_trail() to see what Bob uses most
  • Export valuable procedures: Run astrobob skills sync regularly
  • Clean up periodically: Use forget() for outdated memories

Error Handling

All tools return errors in a consistent format:

{
  "success": false,
  "error": "Error message",
  "error_type": "ConfigError | AstraConnectionError | MemoryNotFoundError"
}

Common Errors

Error Cause Solution
ConfigError Missing environment variables Check .env file, run astrobob doctor
AstraConnectionError Cannot connect to AstraDB Verify credentials, check network
MemoryNotFoundError Memory ID doesn't exist Verify ID with recall()
ValidationError Invalid parameters Check parameter types and ranges
